# Микросервис расчета стоимости доставки

## 1. Стек технологий
* **Core:** Node.js
* **Язык:** TypeScript
* **Фреймворк:** Express
* **Валидация:** Zod
* **Тестирование:** Jest

## 2. Установка и запуск

### Установка зависимостей:
```bash
npm install
```

### Запуск в режиме разработки:
```bash
npm run start
```

### Запуск тестов:
```bash
npm run test
```

## 3. Ответы на вопросы из ТЗ

### 3.1. Проектирование API-эндпоинта

Основной эндпоинт сервиса: `POST /api/calculate`

#### Request Schema
Эндпоинт ожидает JSON-тело запроса со следующей структурой:
* **`cart`** (объект, обязательно): данные о корзине.
  * `weight` (число, обязательно): вес посылки в килограммах (минимум `0.01`).
  * `dimensions` (объект, обязательно): габариты посылки.
    * `length` (число, обязательно): длина в см (минимум `0.1`).
    * `width` (число, обязательно): ширина в см (минимум `0.1`).
    * `height` (число, обязательно): высота в см (минимум `0.1`).
  * `totalPrice` (число, обязательно): общая стоимость товаров в корзине в рублях (минимум `0`).
* **`address`** (объект, обязательно): адрес доставки.
  * `city` (строка, обязательно): город назначения (минимум 1 символ).
  * `street` (строка, обязательно): улица.
  * `house` (строка, обязательно): номер дома.

**Пример запроса (Request Body):**
```json
{
  "cart": {
    "weight": 1.5,
    "dimensions": {
      "length": 15.0,
      "width": 10.0,
      "height": 5.0
    },
    "totalPrice": 2500
  },
  "address": {
    "city": "Москва",
    "street": "Тверская",
    "house": "12"
  }
}
```

#### Response Schema
Ответ возвращается в стандартизированном JSON-формате `{ data, error, details }`.

При успешном расчете (HTTP 200 OK):
* **`data`** (массив объектов `DeliveryOption`): список доступных тарифов.
  * `providerName` (строка): название транспортной компании.
  * `optionName` (строка): название конкретного тарифа или типа доставки.
  * `price` (число): стоимость доставки в рублях.
  * `daysMin` (число): минимальный срок доставки в днях.
  * `daysMax` (число): максимальный срок доставки в днях.

**Пример успешного ответа:**
```json
{
  "data": [
    {
      "providerName": "CDEK",
      "optionName": "Склад-Склад",
      "price": 150,
      "daysMin": 2,
      "daysMax": 4
    },
    {
      "providerName": "CDEK",
      "optionName": "Курьер",
      "price": 300,
      "daysMin": 2,
      "daysMax": 4
    },
    {
      "providerName": "Boxberry",
      "optionName": "Отделение",
      "price": 300,
      "daysMin": 1,
      "daysMax": 2
    }
  ]
}
```

При ошибке валидации (HTTP 400 Bad Request):
```json
{
  "error": "Validation failed",
  "details": [
    {
      "code": "too_small",
      "minimum": 0.01,
      "type": "number",
      "inclusive": true,
      "exact": false,
      "message": "Number must be greater than or equal to 0.01",
      "path": ["cart", "weight"]
    }
  ]
}
```

### 3.2. Внутренняя архитектура и расширяемость

Для возможности простого добавления нового провайдера я использовал паттерн **Стратегий** в сочетании с принципом **внедрения зависимостей**. Это позволяет не изменять существующий код при добавлении нового провайдера и гибко управлять каждым провайдером по отдельности.

1. **Интерфейс провайдера (`IDeliveryProvider`):**
   Определяет общий контракт для всех интеграций. Любой новый провайдер обязан имплементировать данный интерфейс.
   ```typescript
   export interface IDeliveryProvider {
     name: string;
     calculate(cart: CartData, address: DeliveryAddress, signal?: AbortSignal): Promise<DeliveryOption[]>;
   }
   ```

2. **Классы провайдеров (`MockCdekProvider`, `MockBoxberryProvider`):**
   Каждая служба доставки инкапсулирует свою внутреннюю логику взаимодействия с внешним API и расчет тарифов в отдельном классе. Смена или отвал одного из провейдеров не влечет за собой гибель остальных.

3. **Оркестратор (`DeliveryService`):**
   Класс `DeliveryService` принимает массив провайдеров через конструктор (Dependency Injection) и не знает о конкретных реализациях провайдеров:
   ```typescript
   export class DeliveryService {
     constructor(private readonly providers: IDeliveryProvider[]) {}
     // ...
   }
   ```

#### Добавление нового провайдера (например, «Яндекс Доставка»):
1. Создать новый файл провайдера (например, `src/providers/yandex/mock-yandex.provider.ts`).
2. Реализовать класс `MockYandexProvider`, имплементирующий `IDeliveryProvider`.
3. Добавить его экземпляр в массив провайдеров при инициализации приложения в `src/app.ts`:
   ```typescript
   const deliveryService = new DeliveryService([
     new MockCdekProvider(),
     new MockBoxberryProvider(),
     new MockYandexProvider() // Новая интеграция
   ]);
   ```
Существующий код классов `DeliveryService`, `DeliveryController`, роутов и других провайдеров при этом **не изменяется**.

### 3.3. Надёжность и отказоустойчивость

При возникновении высокой нагрузки реализовал следующие решения:

1. **Изоляция сбоев внешних API через `Promise.allSettled`:**
   Параллельные запросы к провайдерам выполняются с помощью `Promise.allSettled` вместо `Promise.all`. Если внешнее API одного из провайдеров недоступно (падает с ошибкой 5xx, ошибкой авторизации или сети), сервис не прерывает работу. Сбойный провайдер логируется в `console.error`, а успешные ответы остальных провайдеров собираются и возвращаются пользователю.

2. **Защита от зависания внешних API Таймауты через `AbortSignal`:**
   Внешние запросы к провайдерам ограничиваются таймаутом (по умолчанию 2000 мс). Для этого используется связка `AbortController` и `AbortSignal` через вспомогательную утилиту `withTimeoutSignal`.
   - Если провайдер не успевает вернуть ответ за 2000 мс, генерируется событие `abort` на переданном сигнале.
   - Реальные HTTP-клиенты (например, `fetch` или `axios`), поддерживающие `AbortSignal`, немедленно обрывают активное сетевое соединение, что предотвращает утечку ресурсов (socket exhaustion) и переполнение пула соединений.
   - Медленный провайдер отбрасывается по таймауту, не задерживая отдачу результатов от более быстрых систем.

3. **Классическая валидация DTO:**
   Использование валидации `Zod` на уровне роутинга отсекает некорректные запросы до выполнения тяжелой бизнес-логики, предохраняя систему от непредвиденных исключений и перегрузок.
